# Unified Object Timeline

A Flow Screen Component that combines records from up to eight different Flow record collections into one configurable table or responsive timeline.

- No Apex, no server-side controller, no package.
- Generic `SObject` property types — each of the eight source slots can hold a different object type, standard or custom.
- Display modes: `table`, `timeline`, or `auto` (responsive — switches to timeline on narrow layouts).
- Search across source, record name, and displayed values; source filtering; date-range filtering; pagination; sortable columns.
- Rich-text/HTML details, expandable in timeline rows.
- Safe by design: a blank source, a missing field, malformed configuration JSON, or removing the Source column all degrade gracefully instead of breaking the component.

The component does not create records, query Salesforce, or discover related records on its own. **The consuming Flow owns data retrieval** and passes the resulting record collections into the component.

## Bundles in this component

| Bundle | Role |
|---|---|
| `unifiedObjectTimeline` | The runtime component you place on a Flow Screen. |
| `unifiedObjectTimelineEditor` | The Flow custom property editor. Not placed directly on a screen — Flow invokes it automatically when you configure `unifiedObjectTimeline` because it's declared as that component's `configurationEditor`. |

Configure the component through the editor. It writes JSON into the runtime component's configuration properties; hand-editing that JSON is possible but not the supported workflow.

## Prerequisites

- Salesforce CLI (`sf`), or an equivalent Salesforce DX deployment workflow.
- A target org and permission to deploy Lightning Web Components.
- My Domain enabled and deployed.
- Lightning runtime enabled for Flows.
- Permission to create, edit, and activate Flows.
- Access (FLS/sharing) to the objects and fields you plan to display.
- One or more Flow record collections containing the records to display.

## Deploy the component source

From a clone of this repository:

```bash
sf org login web --alias my-org

sf project deploy start \
  --target-org my-org \
  --source-dir force-app/main/default/lwc/unifiedObjectTimeline \
  --source-dir force-app/main/default/lwc/unifiedObjectTimelineEditor
```

If the CLI reports an asynchronous deployment, confirm the result with:

```bash
sf project deploy report --target-org my-org
```

Deploying the two bundles is sufficient — there is no Apex class, Flow, custom object, custom field, permission set, or package dependency in the component itself.

## Build the consuming Flow

1. Create or open a Screen Flow.
2. Retrieve the records needed for your use case with Get Records elements.
3. Make sure each source is configured as a **record collection**, not a single record.
4. Add a Screen element and search the component picker for **Unified Object Timeline**.
5. For each source slot you want to use, select the object type and the matching record collection.
6. Configure the visible columns first, in the editor.
7. Configure each source's field mappings into those columns.
8. Choose a date field to sort on, and optionally a fallback date field.
9. Optionally set a label. Leave it blank for no title, or use a Flow text/reference value for a dynamic title.
10. Choose a display mode — `Auto` unless the page layout requires a fixed table or timeline.
11. Choose records per page (default 10).
12. Save, debug, activate, and place the Flow wherever it's needed (record page, quick action, utility bar, etc.).

The consuming Flow is intentionally **not** part of this repository — object relationships, collection names, fields, permissions, and page placement are all org-specific.

## Public Flow inputs

These are the stable `@api` inputs on the runtime component. Treat any rename or type change as a breaking change for consuming Flows.

| Input | Type | Purpose |
|---|---|---|
| `headerLabel` | String | Optional title. Blank hides the title. Static text or a Flow text/reference value. |
| `displayMode` | String | `auto`, `table`, or `timeline`. |
| `pageSize` | Integer | Records per page. The editor offers 5, 10, 15, 20, 25, 50, 100; invalid values are clamped safely at runtime. |
| `columnConfiguration` | String | JSON array defining visible columns. Generated by the editor. |
| `sourceFieldConfiguration` | String | JSON object mapping source slots and fields into columns. Generated by the editor. |
| `source1`–`source8` | SObject collection | The Flow record collections to combine. Each slot can use a different object type. |

The custom property editor is declared in `unifiedObjectTimeline.js-meta.xml` as:

```xml
configurationEditor="c-unifiedObjectTimelineEditor"
```

## Configuration concepts

### Source slots

There are eight independent source slots. Each slot has:

- an object type;
- a Flow record collection of that object type;
- a label shown in the Source column;
- a primary date field, and an optional fallback date field;
- a field mapping for each visible column.

Leave unused slots blank. Custom objects are supported when they exist in the target org and the Flow exposes a collection of that type — the component does not hard-code any organisation-specific custom object into its defaults.

The default configuration uses only standard `Case` fields on source 1, as a minimal working example. Sources 2–8 are blank by default.

### Column definitions

The default columns are:

1. When
2. Source
3. Record
4. Status
5. Type
6. Details

You can rename, reorder, add, or remove columns. Each column has an internal key and a visible header label. **The internal key should not be changed casually once a Flow is configured** — field mappings refer to it.

Supported display types include date, date/time, text, number, currency, percent, boolean, record/link, and rich text/text area/HTML.

The Source column is a convenience column populated from the configured source label. It is not required by the runtime — removing it does not break row creation, filtering, icons, sorting, or the timeline view.

### Field mappings

Each source maps its own object fields into the configured column keys. A column can accept one field or multiple fields; multi-field columns support a configurable separator (including a line break) and can optionally include field labels.

Examples:

- Case: `Type` + `Reason` in the Type column.
- Case: `ClosedDate`, plus your own outcome/score fields, in Details.
- Campaign Member: the campaign name in Record or Details.
- Task: `Subject`, `Status`, `Type`, `TaskSubtype`, `Description`.
- Email Message: `Subject`, `FromAddress`, `TextBody`, falling back to `HtmlBody`.
- Any custom object: its own Name, status, type, and detail fields.

Rich-text details should be configured as a rich-text display column. In table mode the value is summarised; in timeline mode the row can be expanded to show the formatted content.

### Dynamic labels

`headerLabel` (shown as **Label (optional)** in the editor) controls the title. Leave it blank for no title. For a dynamic title, use a Flow reference selected through Flow Builder rather than typing an element reference by hand.

## Troubleshooting

| Symptom | Likely cause |
|---|---|
| Component shows no rows | Check the source is a **collection** variable (not a single record), and that it's populated before the screen renders. |
| A field's value never shows | Confirm the running user has field-level access, and that the field name in the mapping matches the API name exactly (custom fields need the `__c` suffix). |
| Configuration reverted to defaults unexpectedly | The configuration JSON was likely malformed — re-open the editor and re-save each source/column rather than hand-editing the underlying JSON. |
| Icons missing for a custom object | Only objects the component recognises get an icon; unrecognised objects render without one — this doesn't affect data, sorting, or filtering. |
| Timeline doesn't switch layout on a narrow screen | Confirm `displayMode` is set to `auto`. `table` and `timeline` are both fixed modes. |

## What this component does not do

- It does not include a Flow, Apex class, custom object, or test data — those are yours to build.
- It does not query Salesforce or discover related records — the consuming Flow supplies the data.
- It is not distributed as a managed or unlocked package — deploy the source directly.
